Problems

目前搜狗使用的Passport库 http://s.account.sogoucdn.com/u/api/sogou.js 是从 搜狐Passport 修改而来，目前具有以下等一干问题：

  1. 代码冗余----掺杂了许多搜狗不使用的功能代码；
  2. 接口混乱----API 函数的使用十分仰赖经验，难以上手；
  3. 功能耦合----通行证核心功能与对话框 DOM 操作混在一起；
  4. 源码遗失----该 lib 的源码搜狐未能提供，目前仅存压缩后的代码

由于该 lib 不能给搜狗提供统一的登录对话框样式，因此各个产品线不得不拷贝固定的 HTML 代码，按照固定的模式初始化 Passport 功能，造成了各式各样的风格。

product style

这种不受 Passport 控制的部署对以后的升级维护很不利，同时也给各个产品线带来了额外的不必要的工作量。

Work

重构技术路径：

  1. 分离出核心的功能，为需要最大灵活性、自定义性的产品线提供最基本的登录登出功能；
  2. 不必要的功能以集成插件形式存在，预计集成插件代码量不大，为简单起见，可以打进核心包；
  3. 统一的登录对话框以分离的插件形式按需引入，并开放自定义功能

Progress

初步的构建基本完成，下面的设计可能存在的不合理之处，以及可能造成混淆的 case，已经以 Question 的形式提出以供讨论:

由于各产品线对于跨域页面文件 jump.html 的部署已经十分广泛，为兼容及重用，我们仍然使用 PassportSC 作为入口对象名：

<script type="text/javascript" src="dist/@version@/js/passport-core.js">
</script>
<script type="text/javascript">
    window.PassportSC
</script>

你也可以提前定义 PassportSC 对象并异步引入库文件，在这种情况下，其 onApiLoaded 函数会被回调：

<script type="text/javascript">
var PassportSC = {
    onApiLoaded:function(){
        alert('library loaded');
        //Do your passport work here
    }
};
loadSync('dist/@version@/js/passport-core.js');
</script>

初始化函数必须首先调用：

PassportSC.init(options);

options 参数需要包括:

  1. appid: null
  2. redirectUrl: null

这里有必要简述一下 JavaScript 登录原理，即利用表单向 https://account.sogou.com 提交来传递登录验证信息，结果会通过嵌入的 iframe 中的同域的 jump.html 页面传回主页面。因此一个不可见的表单和 iframe 需要构建并加入DOM树中。

表单和 iframe 可以共存于一个不可见元素中，之前需要主动提供该元素，现在由程序自动构建，并利用 CSS 保证其不显示在视窗中，这样是否合理，有无不安全之处？同时登录登出功能共享该元素是否合理？

新的 API 使用事件获取异步接口的结果：

PassportSC.on('loginsuccess loginfailed logoutsuccess',function(evt,data){});

登录接口（异步）：

PassportSC.login(username, password, vcode, autoLogin);

登出接口（异步）：

PassportSC.logout(); 由于登出调用并无失败返回，因此 logoutsuccess 事件是否有意义，在其事件回调内的代码是否安全？

获取用户id，该值应为邮箱格式，依赖于登录态 cookie：

PassportSC.userid();

第三方登录：

PassportSC.login3rd(provider,display,redirectUrl); 第三方登录在使用 window.open 打开弹出窗口需要依赖一个同域下的回调文件，即 redirectUrl，该值应该在 Passport.init 中传入还是应该在 PassportSC.login3rd 中传入？

以上即是 Passport 的核心 API，称 core，下面的功能以集成插件的形式存在，做到完全不向 core 中注入与核心功能无关的功能。

PassportSC.draw(options);

该 API 即提供了统一构建登录对话框的功能，无需手动调用 login 等接口，但它不限定对话框的样式以及脚本逻辑，事实上，它一般用于加载未来 Passport 官方定义的对话框样式及功能，各个产品线完全可以忽略该接口，使用 core 来定义自己的对话框。

对于对话框的不同风格，我们称之为 skin。PassportSC.draw 即负责加载线上 css 文件与该 skin 的 js 脚本文件。它们之间使用称之为 skin_loaded 的事件进行通信，符合该协议规则的任意 skin 都可以被 PassportSC.draw 加载使用。

通过这样的简单设计，我们独立了 Passport 与 skin，它们相互之间的更新升级都不会也不应该影响到对方。

是否需要考虑未来可能存在较大的集成插件对仅使用核心功能的产品带来的影响？ 是否有必要定义统一的搜狗登录框样式？

更多的集成插件仍在丰富中...

PassportSC(options);
